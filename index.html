Generative AI

What is Generative AI

Not vendor hype — a precise understanding of what LLMs are, how they were built, and exactly what they can and cannot do.

🧠
Statistical Pattern Machines

LLMs are autoregressive transformers trained on internet-scale text to predict the next token in a sequence. Through billions of parameters, they compress linguistic patterns, factual associations, and reasoning heuristics. Not rule-based. Not a search engine. A learned probability distribution over language.

🌟
In-Context Learning

The defining capability: LLMs adapt to new tasks at inference time, purely from examples and instructions in the prompt — with no model weight updates. This is why prompting is a genuine engineering discipline. Same model, radically different outputs depending on how you frame the input.

⚠
Core Failure Modes

Hallucination — confident fabrication when pattern-matching fails. Knowledge cutoff — weights are frozen at training time. Organisation-blindness — no awareness of your data. These three failures are precisely what RAG is designed to address.

// LLM inference — what happens when you send a prompt

✓  What LLMs do well

•Natural language understanding & generation at scale
•Code synthesis, debugging, and review across languages
•Structured extraction from unstructured text
•Summarisation, translation, format conversion
•Following complex multi-constraint instructions reliably

✗  Where they fail without RAG

•Anything after training cutoff — knowledge is frozen in weights
•Your organisation's data — it has never seen it
•Verifiable facts — fills gaps with confident fabrication
•Citing sources — parametric memory has no provenance
•Precision on domain-specific terms without explicit context

Transformer

Transformer
A neural network architecture built on self-attention. Introduced in 2017, it processes all tokens in parallel, learning which parts of the input matter for each output — enabling the scale of modern LLMs.
🔠
Tokenisation
Click to reveal
Tokenisation
Splitting text into sub-word units called tokens (~¾ of a word each). Models never see raw characters — only token IDs. Context window limits are measured in tokens, not words or characters.
🅾
Embedding
Click to reveal
Embedding
A dense numeric vector representing a token or document in high-dimensional space. Semantically similar words cluster together. This is how models measure meaning — through geometry, not rules.
👀
Self-Attention
Click to reveal
Self-Attention
A mechanism that lets every token weigh every other token in the context window. This is how the model links pronouns to nouns, resolves ambiguity, and tracks context across long passages.
📊
Parameters
Click to reveal
Parameters
The numerical weights learned during training. GPT-4 has ~1 trillion. They encode all the model's knowledge. Once trained, they are fixed — the model cannot learn new facts without retraining or RAG.
⚡
Inference
Click to reveal
Inference
Running a trained model to generate output — what happens every time you send a prompt. Distinct from training, which updates weights. Inference is fast and cheap; training is slow and very expensive.
🌡
Temperature
Click to reveal
Temperature
A sampling parameter controlling output randomness. At 0 the model picks the highest-probability token every time (deterministic). At 1+ it samples more freely — more creative, less precise.
🔌
Context Window
Click to reveal
Context Window
The maximum tokens a model can see at once — input and output combined. GPT-4o: 128K tokens. Everything in the window shapes every token generated. Larger = richer context = better answers.
🌟
RLHF
Click to reveal
RLHF
Reinforcement Learning from Human Feedback. Human raters score model outputs, and the model is fine-tuned toward preferred responses. This turns a raw text predictor into a helpful assistant like Copilot.
🔢
Vector Search
Click to reveal
Vector Search
Finding documents by semantic similarity, not keyword matching. A query is embedded into a vector, then the index returns the nearest document vectors by cosine distance. The foundation of every RAG system.
🏢
Foundation Model
Click to reveal
Foundation Model
A large pretrained model (GPT-4o, Claude, Gemini) trained on internet-scale data, then adapted for specific tasks. Called “foundation” because other products are built on top of them.
💬
Prompt
Click to reveal
Prompt
The full input sent to the model — typically a system message (hidden instructions) plus the user message. Prompt engineering is the discipline of crafting inputs to reliably elicit the outputs you need.
Under the Hood

How Gen AI Works

The transformer architecture in enough depth to reason about why prompts work and why they fail.

Step 01
🔠
Tokenisation

Text splits into sub-word tokens (~¾ of a word each). "Power Automate" becomes ["Power"," Autom","ate"]. The model never sees raw characters — only token IDs mapped to learned embedding vectors.

Step 02
🅾
Embeddings

Each token ID maps to a dense vector (12,288 dimensions in GPT-4). Semantically similar tokens cluster in this space. This is where meaning enters the computation — not rules, pure geometry.

Step 03
👀
Self-Attention

Every token attends to every other token in context. Multi-head attention learns which tokens matter to each other — capturing long-range dependencies, pronoun resolution, and contextual meaning shifts.

Step 04
🔋
Feed-Forward Layers

After attention, each position passes through FFN layers encoding learned knowledge. Most factual knowledge appears to be stored here in the model weights — not in the attention mechanism.

Step 05
🎲
Next Token Sampling

The final layer outputs a probability distribution over ~50k tokens. The model samples the next token, appends it, and repeats. This autoregressive loop is generation.

Step 06
🌟
RLHF Alignment

Raw pretrained models output chaotic continuations. RLHF trains the model to follow instructions and be helpful — producing the clean assistant behaviour you see in Copilot. Temperature and system prompts shape outputs at inference time.

// Key inference parameters you control via prompting
Context Window

All text the model sees at once. GPT-4o: 128K tokens (~96K words). Everything inside influences every token generated. Paste context generously.

Temperature

Sampling randomness. 0 = deterministic. 1+ = creative. Copilot uses ~0.7 for chat, lower for code. Shapes the probability distribution, not a creativity dial.

System Prompt

Hidden instructions prepended before your message. How Copilot 365, custom agents, and RAG systems configure model behaviour, role, constraints, and knowledge sources.

In-Context Learning

Examples and instructions in the prompt shift model behaviour with no weight update. Few-shot examples, personas, format schemas — all work through this mechanism.

RAG

What is RAG

Retrieval-Augmented Generation — the architectural pattern that addresses every major LLM failure mode for enterprise use.

The Core Problem RAG Solves

LLMs have static parametric knowledge — baked into weights at training time, containing nothing beyond the cutoff date and nothing specific to your organisation. Ask GPT-4o "What is the error handling procedure for our SAP sync flow?" and it will confidently fabricate a plausible-sounding but factually wrong answer for your context. This is the fundamental enterprise AI problem.

RAG separates retrieval of relevant information from language generation. At query time, the system fetches the most semantically relevant content from an external knowledge store — your SharePoint, Celonis reports, runbooks — then injects that content into the model's context alongside the question. The model generates grounded in actual, specific, up-to-date information. Every answer is citable. Every claim is verifiable. This is the architecture behind every trustworthy enterprise AI system in production today — including Microsoft Copilot 365.

// Parametric LLM vs RAG-augmented LLM

❌ Without RAG — Parametric LLM

•Frozen knowledge — nothing post-cutoff, nothing org-specific
•Hallucination risk — fills factual gaps with confident fabrication
•Organisation-blind — knows nothing about your bots, runbooks, or Celonis data
•Non-citable — no provenance; impossible to audit in a GxP environment

✓ With RAG — Grounded Generation

•Live knowledge — retrieves from documents updated today
•Grounded answers — model instructed to answer only from retrieved context
•Organisation-aware — your IA SharePoint PDD, SDD, runbooks become queryable
•Source-cited — every answer references the exact document and section

Chunking Strategy

Semantic chunking (split at headings, paragraphs) outperforms fixed-character splitting. Overlap (15-20%) prevents context loss at boundaries. Chunk size trades precision (small) vs context (large).

Semantic SplitSliding Window512-1024 tokens
🔢
Embedding Models

Identical model at index and query time is mandatory. text-embedding-3-large (3072-dim) outperforms ada-002 on most benchmarks. Never mix models across phases.

ada-002text-embedding-3Cosine Similarity
🔍
Hybrid Search

Pure vector search misses exact-match queries (error codes, IDs). Pure BM25 misses semantic variants. Hybrid via RRF captures both. Azure AI Search supports this natively. Always use hybrid in production.

BM25RRF FusionAzure AI Search
🎯
Reranking

Vector retrieval gets you approximately right chunks. A cross-encoder reranker (Azure Semantic Ranker) re-scores with full query-document attention — dramatically improving precision. The latency cost is worth it.

Cross-EncoderSemantic RankerTop-K

Copilot 365

Copilot 365 as a RAG System

Copilot 365 is not magic — it is a production RAG pipeline over Microsoft Graph. Knowing the architecture tells you exactly how to prompt it effectively.

How Copilot 365 Retrieves

Every Copilot 365 query triggers retrieval against the Microsoft Graph — the unified API indexing your emails, Teams messages, SharePoint documents, OneDrive files, Calendar, and meeting transcripts. Microsoft uses a semantic index built on Azure AI Search to make all of this retrievable by meaning, not just keyword.

Retrieved content is injected into GPT-4o's context window alongside your prompt. The model generates grounded in your real organisational data — cited, recent, and specific to you.

What this meansCopilot is not guessing about your Invoice Automation — it is reading your actual emails, SharePoint PDD, and Teams conversations. Answer quality is directly proportional to the quality and recency of your M365 content.

Prompting Copilot Effectively

Since Copilot does RAG over Graph data, prompts should explicitly reference real artefacts — this signals which data to retrieve and dramatically improves answer quality.

Weak — generic retrievalGive me a status update on the invoice automation.
Strong — targeted retrievalBased on the PDD in our IA SharePoint Bot Documentation library, my email thread with Finance this week, and action items from Monday's Teams standup — summarise project status, top 2 risks, and the one thing I need to do today.

🎭
Role — Set the Expert Persona

System prompt priming shifts the model's distribution toward expert-level output. "Act as a senior Power Automate architect" activates relevant knowledge clusters. Always match the role to the specific domain of the task.

ExampleAct as a senior Celonis EMS expert with deep SAP P2P event log knowledge and PQL optimisation experience...
🎯
Task — One Precise Action

One action verb per prompt. Compound tasks dilute attention. Chain them sequentially. Use precise verbs: generate, extract, refactor, debug, compare, classify. Not vague ones like "help me with" or "can you look at."

Example...design a Power Automate flow triggered on new SharePoint Invoices item, routing approval when amount exceeds $5,000...
📎
Context — Maximise Relevance

Paste the full error message. Paste the code block. Paste Celonis KPI values. Paste the meeting transcript. The model attends over everything — more signal means less noise. Do not paraphrase what you can paste verbatim.

Example...standard connectors, SharePoint Online, SAP S/4HANA via HTTP, E3 licensing. Full error: [paste]...
📐
Format — Lock the Schema

Unspecified format gives unpredictable structure. Define exactly: numbered steps with connector name and expression, JSON with keys A B C, executive summary under 200 words. The model conforms reliably to explicit schema instructions.

Example...numbered steps: connector | operation | expression (if applicable) | error condition.
🚫
Constraints — Hard Boundaries

Exclusions prevent the model defaulting to its highest-frequency generic patterns. "No premium connectors", "avoid technical jargon", "standard Celonis objects only". Constraints narrow the solution space to what is actually viable.

Example...standard connectors only, max 12 steps, must handle duplicate record exception, no hardcoded values.
🔄
Iterate — Multi-Turn Refinement

LLMs are stateful within a conversation. The first response is a well-informed draft. Follow-up prompts build on prior context. Chain atomic refinements: "Now add retry logic", "Rewrite for non-technical audience", "Add test scenarios for null input."

Follow-up patternsNow add... · Rewrite [section] for... · What if [edge case]? · Show 3 alternatives · Generate test cases

Advanced Techniques

Advanced Prompt Engineering

Techniques that separate consistent, production-quality outputs from variable, unpredictable ones. Engineering patterns, not tricks.

Chain-of-Thought

Instruct the model to reason step-by-step before the final answer. "Think through this step by step" activates more deliberate computation and reduces errors on multi-step reasoning, debugging, and architectural decisions.

Pattern...before designing the flow, reason through the data dependencies and potential race conditions step by step, then provide the final design.
Few-Shot Exemplars

Provide 2-3 input-output pair examples before your actual request. The model infers the pattern and dramatically improves format consistency. Powerful for custom document formats like Baxter's PDD template and classification tasks.

PatternHere are two correctly formatted user stories from our last project: [example 1] [example 2]. Now write user stories for this: [requirement].
Self-Consistency Sampling

For high-stakes outputs, request 3 independent analyses and identify convergence. "Give me 3 independent analyses and identify where they agree." The majority answer across samples is more reliable than any single answer.

PatternAnalyse this architecture from 3 angles: performance, security, and maintainability. Then identify where all three agree on the biggest risk.
Persona + Constraint Stacking

Combine a specific expert persona with hard constraints to narrow the probability distribution precisely. This is far more effective than a generic request — it activates the right knowledge and eliminates unsuitable solutions in one prompt.

PatternAct as a GxP-aware automation architect. Design only to Baxter IT-approved connector list. Every decision must include a GxP risk justification.
Output Schema Locking

Define the output as a rigid schema — JSON structure, table with exact columns, numbered sections with defined headings. Locked format channels creativity entirely into content quality rather than structural decisions. Produces consistently parseable outputs.

PatternReturn as JSON: {{"risk": string, "severity": "High|Medium|Low", "mitigation": string}}. No preamble or explanation outside the schema.
Anti-Hallucination Grounding

For accuracy-critical tasks, add an explicit uncertainty instruction: "If you are not certain about a specific API endpoint or technical detail, say so rather than guessing." Shifts sampling toward admitted uncertainty over confident fabrication.

Pattern...if uncertain about any specific SharePoint REST endpoint or PA expression syntax, state your uncertainty rather than guessing.

Patterns That Work

✓Paste the full error message — never summarise it
✓Specify exact output format (table columns, JSON schema, step structure)
✓State what NOT to include as explicitly as what to include
✓Chain follow-up prompts — refine in steps, not one shot
✓Ask for reasoning before the final answer on complex tasks
✓Request 3 alternatives before committing to an approach
✓Ask for test cases as a dedicated follow-up prompt

Failure Patterns to Avoid

✗Accepting the first output without iteration
✗Asking 5 different things in a single prompt
✗Using vague verbs: "help me", "make something", "improve this"
✗Paraphrasing error messages instead of pasting verbatim
✗Not specifying audience for communication drafts
✗Trusting specific technical details without verification
✗Starting a new chat for every follow-up (destroys context)
